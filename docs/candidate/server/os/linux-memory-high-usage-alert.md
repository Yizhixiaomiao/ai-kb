## 适用范围

- 适用于服务器、业务机器、Kubernetes 节点内存使用率超过 90%/95%/97% 的告警。
- 适用于规则名称包含“内存使用率超过95%”“内存使用率达到97%”“内存压力过大”等场景。

## 常见现象

- 内存使用率持续高于阈值。
- 系统出现 OOM、进程被杀、服务响应慢或容器重启。
- Kubernetes 节点出现 MemoryPressure。

## 判断依据

- 区分 Linux 缓存占用和真实应用内存压力。
- 结合 swap、OOM 日志、进程 RSS、容器限制和业务日志判断。

## 处理步骤

1. 确认告警对象、业务系统、持续时间和是否伴随 OOM 或服务重启。
2. 查看内存、swap、进程占用、容器占用和系统日志。
3. 判断是缓存、正常业务峰值、内存泄漏、批处理任务还是资源限制过低。
4. Kubernetes 场景下查看 Pod requests/limits、节点资源、事件和重启次数。
5. 如确认业务异常，按预案扩容、重启异常实例、回滚发布或转应用负责人分析内存泄漏。
6. 记录高内存进程、处理动作、是否 OOM、恢复时间和后续优化建议。

## 验证方式

- 内存使用率回落并稳定。
- OOM 或重启不再发生。
- 业务访问、日志和监控恢复正常。

## 注意事项

- 不要把 Linux buff/cache 简单等同于内存泄漏。
- 不要直接清缓存或重启服务作为长期方案。
- 涉及生产服务重启、扩容和参数变更时必须确认审批和回退。

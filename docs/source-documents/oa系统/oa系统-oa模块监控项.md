## Sheet1

| 业务层级 | 监控项 |  |

| 中间层 | JVM堆内存使用率>80% | 可能故障：应用存在内存泄漏、内存分配过大、垃圾回收不及时，可能导致后续出现OOM（Out Of Memory）错误，服务崩溃或响应变慢。 |

|  | GC吞吐率<90%,程序异常、内存泄漏 | 可能故障：垃圾回收频繁且耗时，应用大部分时间都在GC，业务处理能力下降，用户体验变差。 |

|  | 5分钟内GC暂停总时长大于1秒时告警 | 可能故障：频繁或长时间的GC会导致应用线程“停顿”，用户请求响应变慢，严重时会造成服务不可用。 |

|  | 活跃的守护线程数量>500 | 可能故障：线程泄漏、死循环、线程池配置不合理，可能导致CPU资源耗尽，应用卡死或崩溃。 |

| 业务层 | 业务可用性<95% | 可用性低于阈值（如95%）时，说明服务健康检查失败率升高，可能服务已不可用或部分实例异常。 |

|  | QPS（吞吐量） | 每个模块差距较大 |

|  | 4XX/5XX接口异常告警 | 某些接口参数错误（4xx）、后端故障（5xx）频发，需重点排查这些接口的调用和依赖。 |
